# Чайка Парк — система онлайн-бронирования лежаков

## О проекте
Веб-сервис для парка «Чайка Парк» (Благищенский), позволяющий гостям онлайн:
- видеть интерактивную схему территории (пляж / бассейн / VIP-зоны)
- видеть статус каждого лежака/шезлонга/кабинки в реальном времени (свободно / занято / забронировано)
- бронировать конкретное место на конкретную дату и время
- оплачивать бронь картой или через СБП (Т-Касса / Тинькофф Эквайринг)
- получать подтверждение и чек в Telegram и/или на почту

## Зоны и объекты
- **Пляж** — обычные лежаки
- **Бассейн** — лежаки у бассейна
- **VIP-зоны** — беседки/кабинки с повышенной ценой и отдельным набором опций

У каждого объекта (`spot`): номер, зона, координаты на схеме (x, y в процентах или пикселях от карты), базовая цена, статус.

## Технический стек
- **Frontend**: Next.js 14+ (App Router), TypeScript, Tailwind CSS
- **Backend**: Next.js API routes (единый репозиторий, без отдельного бэкенда)
- **БД**: PostgreSQL + Prisma ORM
- **Realtime**: Socket.io (статусы лежаков обновляются у всех клиентов без перезагрузки)
- **Платежи**: Т-Касса (Тинькофф Эквайринг) — карты + СБП через единый API (`Init` / `Notification` webhook)
- **Уведомления**:
  - Telegram Bot API (свой бот через @BotFather)
  - Email через SMTP/Resend
- **Инфраструктура**: Docker Compose (app + postgres + nginx), деплой на существующий VPS, SSL через certbot

## Бизнес-правила бронирования
- Бронь всегда привязана к дате + временному слоту (например, "весь день" или почасово — уточнить у заказчика)
- Место переходит в статус `pending` сразу при создании брони (резерв на N минут, например 15), чтобы два человека не забронировали одно и то же одновременно
- Статус `confirmed` — только после подтверждения оплаты через вебхук Т-Кассы
- Если оплата не пришла за N минут — бронь автоматически отменяется, место освобождается (cron/очередь)
- VIP-зоны могут требовать предоплату/депозит — уточнить процент

## Конвенции разработки
- Код и комментарии — на английском, тексты интерфейса — на русском
- Все платёжные операции — только через backend (API routes), никогда не доверять статусу оплаты, пришедшему с клиента
- Вебхук от Т-Кассы должен проверять подпись (Token) запроса перед обработкой
- Все денежные суммы хранить в копейках (integer), не в float
- Используем миграции Prisma, не редактируем БД руками в проде

## Порядок реализации (итерации)
1. Скелет: Next.js + Postgres + Docker Compose + Prisma миграции по схеме
2. UI: интерактивная схема зон с кликабельными лежаками и статусами (сначала на моках, без оплаты)
3. Флоу бронирования: выбор места → форма контактов → создание pending-брони
4. Интеграция Т-Кассы: Init платежа (карта + СБП), вебхук подтверждения, автоотмена pending-броней
5. Telegram-бот и email-уведомления с деталями брони и чеком
6. Realtime-обновление статусов через Socket.io
7. Админка: список броней, ручное управление статусами, статистика
8. Деплой на VPS, SSL, переключение на боевой TerminalKey Т-Кассы

- Нужен ли личный кабинет гостя с историей броней, или бронирование анонимное (по телефону/email)?
- Требуется ли предоплата/депозит для VIP-зон, в каком размере?
